# Glosc Copilot 商业计划书 2.0


## 1. 介绍

Glosc Copilot 是跨平台 AI 桌面助手，核心差异化来自其配套的插件社区（Gloss Store）：

- 对用户：用“可安装的插件能力”把 AI 从聊天变成可复用的工作流与自动化工具。
- 对开发者：提供 SDK 与发布/分发/商业化路径，让 Gloss Store 社区持续产出高价值插件。

我们选择的竞争主线不是“更大模型/更像人”，而是“更强社区/更快落地”：用 Gloss Store 社区插件生态形成网络效应与长期护城河。

## 2. 问题与机会

面向两类核心用户的共同痛点：

### 2.1 办公人群

- 重复性事务占用时间：文档整理、表格处理、信息查找、邮件与会议纪要、跨系统录入。
- 工具碎片化：AI 在聊天窗口里，任务在浏览器/桌面软件里，流程需要人工“搬运”。
- 缺少可复用工作流：同一类任务每次都从零开始描述，难以标准化、难以沉淀。

### 2.2 极客/开发者

- 个人自动化脚本难以产品化：脚本散落、缺少 UI/权限/分发/更新机制。
- AI 工具缺少“可组合能力”：想把提示词、工具调用、系统操作封装成插件，成本高。
- 贡献缺少激励：做出来的能力难以被发现、复用与变现，社区难形成正反馈。

核心判断：AI 助手赛道最终比拼的是“能力供给速度”。社区插件生态可以把能力供给从“团队产能”提升为“社区产能”。

## 3. 解决方案：以插件社区生态为中心

Glosc Copilot 的解决方案由“产品内扩展能力 + 社区机制”两部分组成。

### 3.1 产品内扩展能力（让插件可用、可控、可交付）

- 插件运行与沙箱：标准化插件生命周期、权限与资源隔离。
- 工具调用与任务编排：把常见能力封装为可调用工具（文件/网页/知识库/第三方 API/企业系统等），插件可组合。
- 可观测与可回滚：日志、错误上报、版本管理，降低插件使用与维护成本。
- 跨平台一致性：Windows/macOS/Linux 统一插件接口与行为。

### 3.2 Gloss Store 社区机制（让插件“有人做、有人用、有人付费”）

- SDK 降门槛：Python/TypeScript 等语言支持（以现有技术栈为基础逐步完善）。
- 社区分发：检索、安装、更新、评分、反馈闭环。
- 激励与商业化：免费/付费/订阅/企业授权等多种分发策略与分成规则。

差异化结论：我们不是“又一个聊天助手”，而是“插件社区驱动的能力扩展 + 桌面执行入口”。

## 4. 产品形态与核心功能 (Product)

### 4.1 产品定位

- Glosc Copilot = 桌面入口（执行） + 插件体系（扩展能力） + Gloss Store 社区（分发与协作）

### 4.2 核心功能模块（按主线收敛）

| 模块               | 面向对象     | 价值                 | 核心输出                    |
| ------------------ | ------------ | -------------------- | --------------------------- |
| 插件体系（产品内） | 开发者/团队  | 降低开发与维护成本   | SDK、权限、运行时、版本管理 |
| Gloss Store 社区   | 全体用户     | 发现与复用能力       | 安装/更新/评分/分类/搜索    |
| 任务执行层         | 办公人群为主 | 把插件变成可交付结果 | 一键工作流、可复用模板      |
| 组织与治理（企业） | 办公团队/IT  | 可控可管             | 企业插件源/白名单/审计      |

### 4.3 典型场景（用“插件”描述能力）

- 办公：会议纪要插件、周报生成插件、合同要点抽取插件、表格清洗插件、知识库问答插件。
- 开发：Issue 生成/分解插件、PR 审阅辅助插件、代码片段/脚手架插件、日志分析插件。

## 5. 市场与竞争 (Market & Competition)

### 5.1 市场机会

AI 助手与生产力工具市场整体空间大；但我们的切入点更聚焦：

- 以“桌面执行 + 工作流复用 + 插件供给”切入，而不是泛对话。
- 用社区扩张能力边界，用产品工程效率对抗大厂模型优势。

（注：如需对外融资材料，可进一步补充可核验的数据来源与引用口径；本计划书保留原有量级判断，但不以单一数字作为核心论证。）

### 5.2 目标用户

- 极客/开发者：高频创造者与插件供给方，贡献工具与工作流，推动生态增长。
- 办公人群：高频消费者与付费主力，用插件直接产出“可交付成果”。

### 5.3 竞争格局

- 通用大模型/聊天助手：强在通用问答，但弱在任务落地与可复用工作流沉淀。
- 垂直效率工具：强在单点体验，但弱在可扩展性与生态供给。

我们的策略：用“插件生态 + 分发 + 治理”构成社区护城河，让能力供给随社区规模非线性增长。

## 6. 增长策略

增长以“插件供给”驱动“用户增长”，再反哺“插件供给”。

### 6.1 开发者增长（供给侧）

- 提供高质量示例插件与模板：降低从 0 到 1 成本。
- 文档与工具链：脚手架、调试器、权限声明、发布流程。
- 激励机制：榜单、推荐位、分成与企业采购对接。
- 使用AI 一句话构建插件：降低技术门槛，吸引更多非专业开发者参与。

### 6.2 办公人群增长（需求侧）

- 从“可见成果”切入：一键生成纪要/周报/分析报告等，强调 ROI。
- 组织内传播：团队模板、共享插件列表、可复制的最佳实践。
- 轻量上手：默认插件包覆盖 80% 高频场景。

### 6.3 冷启动策略

- 先做“产品级必需插件”：文件处理、网页摘录、表格清洗、知识库等通用能力。
- 再做“行业/岗位包”：运营、销售、HR、研发等。

## 7. 商业模式

商业化围绕“订阅 + Gloss Store 社区分成 + 按需付费”展开，并以生态作为增长飞轮。

### 7.1 收入来源

1) 订阅服务（个人/团队）

- 专业版：更高配额、更强工作流、团队模板与同步等。

2) Gloss Store 社区分成（生态分成）

- 插件/工作流/模板的付费下载或订阅分成（示例口径：社区抽成 30%，可按阶段调整）。

1) 按需付费（低频用户）

- 按使用量计费的高级功能（示例口径：超出免费额度后，按任务次数或数据量收费）。

### 7.2 价格与转化

- 免费版：覆盖基础插件与轻量工作流，承担获客。
- 专业版：以“节省时间/减少返工”的成果定价，目标转化率可在试运营后校准。

## 8. 发展路线图

路线图以“产品可用 + 社区可增长”优先级排序。

Q1 · 插件体系 MVP（服务于 Glosc Copilot）

- 插件运行时、权限模型、基础 SDK
- 核心插件包（办公与开发各 3–5 个高频场景）

Q2 · Gloss Store 社区上线与增长

- 社区分发能力（发布/检索/安装/更新/评价）
- 开发者工具链完善（脚手架、调试、CI 发布流程）

Q3 · 组织化与治理

- 团队模板、共享与协作
- 企业治理能力（私有源/白名单/审计）

Q4 · 产品与生态规模化

- 更强的任务编排与可观测
- 生态运营体系（推荐、榜单、激励、企业采购通道）

## 9. 核心竞争优势

### 9.1 Gloss Store 社区插件生态

- 能力供给来自 Glosc Copilot（核心插件）与社区贡献共同增长
- 通过分发与激励形成网络效应与长期壁垒

### 9.2 插件体系工程能力

- 权限/沙箱/版本/可观测，保障插件“可用、可控、可维护”

### 9.3 跨平台一致性

- 统一接口降低插件开发成本，扩大可服务用户面

## 10. 融资资金分配

- 产品研发（插件体系+核心插件）：55%
- 开发生态与社区运营（文档、模板、激励、市场运营）：20%
- 市场推广（以插件成果案例驱动）：15%
- 基础运营与合规/安全：10%

## 11. 关键指标 (KPIs)

围绕“生态供给 → 使用 → 付费”的链路定义指标。

- 供给侧：月活开发者数、月新增插件数、插件更新频次、Top 插件留存
- 需求侧：插件安装转化率、活跃插件数/人、任务完成次数、办公场景净推荐值（NPS）
- 商业化：付费转化率、ARPU、Gloss Store GMV、企业线索转化率
- 效率：支持成本（每千次任务支持工时）、插件故障率与恢复时间
